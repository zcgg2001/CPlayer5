# CPlayer5 基础改进设计

## 目标

在保持纯静态部署、现有视觉效果和上游同步能力的前提下，完成以下三个阶段：

1. 修复安全与正确性缺陷。
2. 提升网络请求和 PWA 缓存的可靠性。
3. 补齐关键无障碍与移动端交互能力。

本轮不进行框架迁移，不重写播放器，不对 `index.html` 做大规模模块化拆分。

## 全局约束

- 运行时继续使用 HTML、CSS 和原生 JavaScript，不引入 React、Vue 或其他前端框架。
- 项目仍可通过任意静态 HTTP 服务器直接运行。
- 每一阶段使用独立功能分支和 Pull Request。
- 每个阶段开始前先获取并合并最新 `upstream/main`。
- 尽量保持 DOM 结构、视觉设计、播放行为和本地歌单格式兼容。
- 不缓存音频流，不改变第三方 API 的业务用途。
- 不在本轮处理字体体积、完整模块化或视觉重设计。

## 当前风险

### 不可信 HTML 注入

歌曲名称和歌手名称来自远程 API 或用户导入的歌单。桌面搜索、移动端播放列表和移动端搜索结果中存在将这些字段直接插入 `innerHTML` 的代码，可能执行恶意 HTML。

### Service Worker 分支不可达

当前 `music.126.net` 请求先被音频直连规则捕获，后面的封面图片缓存规则无法执行。字体缓存数组使用了不存在的文件名并且没有参与安装缓存。

### 网络错误处理分散

API 请求没有统一的超时、HTTP 状态检查或响应结构验证。部分请求失败后只记录控制台信息，用户无法区分超时、服务异常和无结果。

### 无障碍语义不足

视口配置禁止缩放。多个输入框没有可访问标签，部分图标按钮没有可访问名称，Toast 没有实时播报，模态层缺少对话框语义和焦点管理，部分移动端触控目标小于 44 像素。

## 阶段一：安全与正确性

### 安全渲染

- 所有来自 API、导入文件和缓存的歌曲名称、歌手、专辑等文本使用 `textContent` 或显式创建的文本节点。
- `innerHTML` 仅保留给完全静态且由仓库维护的图标或骨架模板。
- 封面地址和音频地址只接受 `https:`、`http:`、`blob:` 和项目已有的 `data:image/` 占位资源。
- 无效 URL 使用本地占位图或显示可理解的播放错误，不把未经验证的值写入 DOM 属性。

### 重复与死代码

- 合并重复的 `plusready` 初始化，保证返回键和 Wake Lock 只注册一次。
- 删除确认没有调用方的音频标准化、滚动和背景更新函数。
- 保留仍被桌面端或移动端调用的兼容包装函数。

### 验证

- 使用包含 HTML 标签和事件属性的恶意歌曲名称验证不会生成可执行节点。
- 验证普通远程歌单、本地 JSON、`playlist.js` 和搜索结果仍可显示和播放。
- 验证 H5+ 环境初始化函数只绑定一次。

## 阶段二：网络与缓存

### 统一请求行为

新增 `js/http.mjs` 请求辅助模块，由播放器和歌单下载工具共同导入，统一实现：

- 默认 10 秒超时，调用方可以显式覆盖。
- 检查 `response.ok` 后再解析 JSON。
- JSON 解析失败、HTTP 错误、超时和用户主动取消使用不同错误类型。
- 调用方将错误转换为面向用户的中文提示，同时保留控制台诊断信息。
- 搜索输入变化或重复提交时取消已经过期的请求，避免旧结果覆盖新结果。

### API 数据验证

- 搜索、歌曲、歌词和歌单响应分别经过最小结构检查。
- 缺少必要字段的数据被忽略，不进入播放列表和缓存。
- 未知可选字段使用明确默认值，避免在渲染阶段散落兼容判断。

### Service Worker

- 将封面图片判断放在通用 `music.126.net` 直连规则之前。
- 音频和 API 继续采用网络直连，不写入 Cache Storage。
- 本地核心资源采用缓存优先并在后台更新。
- 封面采用缓存优先，设置容量或过期清理策略，避免无限增长。
- 删除失效且未被使用的字体缓存数组；字体继续由运行时本地资源缓存处理，本轮不加入安装预缓存。
- 安装缓存失败时保留清晰日志，不把全部安装错误静默吞掉。
- 提供最小离线页面回退；离线时明确说明搜索和播放需要网络。

### 验证

- 测试 HTTP 4xx、5xx、无效 JSON、超时和断网场景。
- 验证封面请求进入缓存分支，音频和 API 不进入缓存。
- 验证新 Service Worker 激活后清理旧版本缓存。
- 验证离线打开已缓存页面不会白屏。

## 阶段三：无障碍与移动端

### 页面和表单

- 移除 `maximum-scale=1.0` 和 `user-scalable=no`。
- 为搜索、歌单 ID、文件导入和音量控制补充可见标签或 `aria-label`。
- 图标按钮统一提供 `aria-label`，状态按钮同步更新 `aria-pressed` 或可访问名称。
- Toast 使用 `role="status"` 和 `aria-live="polite"`；错误提示使用 `role="alert"`。

### 模态层和焦点

- 欢迎页、设置面板和移动端底部抽屉使用 `role="dialog"`、`aria-modal="true"` 和指向可见标题的 `aria-labelledby`。
- 打开时将焦点移到标题、首个输入或主要操作。
- 关闭时恢复到触发按钮。
- 支持 Escape 关闭非阻塞对话框，并避免焦点移动到被遮挡的页面内容。

### 触控与动态效果

- 所有主要触控目标至少为 44 x 44 像素，优先扩大点击区域而不是放大图标。
- 为手势操作保留可见按钮替代方案。
- 尊重 `prefers-reduced-motion: reduce`，关闭非必要背景、旋转、弹性滚动和过渡动画。
- 页面进入后台时暂停 WebGL 和 Canvas 动画；恢复可见时继续。
- 保留移动端安全区和 `100dvh` 布局，不引入横向滚动。

### 验证

- 使用键盘完成搜索、播放、打开和关闭设置以及导入歌单。
- 在 390 x 844、768 x 1024 和 1440 x 900 视口检查布局和触控目标。
- 验证浏览器缩放、屏幕阅读器名称、Toast 播报和减少动态效果设置。

## 测试策略

- 单元测试覆盖文本安全渲染、URL 校验、请求错误分类和 API 数据归一化。
- Service Worker 测试覆盖请求分类、缓存命名和旧缓存清理。
- 浏览器冒烟测试覆盖欢迎页、桌面端、移动端、歌单导入和错误提示。
- 每个 PR 继续运行现有静态资源检查。
- 测试先于对应实现提交，确保回归测试能够在修复前失败、修复后通过。

## 交付顺序

1. `fix/security-correctness`
2. `fix/network-pwa-reliability`
3. `fix/accessibility-mobile`

每个 PR 独立可运行、可回退，并在合并前完成浏览器验证。第三阶段完成后重新评估模块边界，再单独设计模块化方案。
