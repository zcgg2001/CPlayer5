# 音乐下载与音质选择设计

## 目标

为网页端播放器增加单曲下载能力。用户可从搜索结果、新歌速递和当前播放栏进入同一下载流程，选择音质后下载当前歌曲。项目保持 MIT、静态部署和 Vanilla JavaScript 架构，不新增依赖或服务端代理。

## 用户交互

1. 搜索结果与新歌速递的每行增加一个可访问的“下载”图标按钮；桌面播放栏为当前歌曲增加同一入口。
2. 点击入口打开一个原生 `dialog`。对话框显示歌曲名与歌手，并提供标准（128K）、极高（320K）、无损（FLAC）、Hi-Res、超清母带五个音质选项。
3. 点击确认后禁用重复提交并显示下载状态。最近一次选择保存在 `localStorage`，下次打开时作为默认值。
4. 成功后浏览器保存文件，文件名为安全化的“歌曲名 - 歌手 - 实际音质.扩展名”。对话框关闭并以既有 Toast 提示实际下载音质。

## 音质解析与降级

下载复用 `ChKSzAPI` 的 `163_music?id=<id>&level=<level>` 接口，不复用播放中的 URL，保证每次下载都按用户所选档位解析。

解析顺序从所选档位向低档位降级：

- `jymaster` → `hires` → `lossless` → `exhigh` → `standard`
- `hires` → `lossless` → `exhigh` → `standard`
- `lossless` → `exhigh` → `standard`
- `exhigh` → `standard`
- `standard` → `standard`

每次请求都使用现有媒体 URL 安全校验。若所有档位都没有有效 URL，显示“该歌曲暂无可下载音质”，不创建下载任务。

## 下载执行与错误处理

浏览器使用 `fetch` 读取已校验的音频 URL，再用 `Blob` 与临时对象 URL 创建下载链接；这样可以生成可读文件名且不暴露不可信 URL 到 DOM。下载过程不写入 Service Worker 或 IndexedDB，避免意外缓存大型音频。

若响应失败、内容为空、网络中断或浏览器的跨域策略阻止读取，保留对话框并显示具体失败提示。下载不会影响正在播放的音频、播放队列或音质偏好。

## 结构与无障碍

下载状态和选中歌曲由一个小型的下载控制器管理。它依赖现有 `ChKSzAPI`、`normalizeMediaUrl` 和 Toast，向三个 UI 入口提供同一 `open(song)` 行为。

`dialog` 使用标题关联、关闭按钮、单选音质控件、明确的确认按钮与 `aria-live` 状态；关闭后焦点返回原下载按钮。图标入口均有可读的 `aria-label`。

## 验证范围

- Node 单元测试：选定音质的解析顺序、可用 URL 的选择、文件名安全化与媒体 URL 拒绝行为。
- 静态标记测试：三个下载入口、对话框语义、音质选择和无障碍名称。
- 启动行为测试：下载控制器初始化、与现有搜索/新歌/当前歌曲数据的连接。
- 完整 Python、Node 与静态站点检查；浏览器实测一次正常下载与一次自动降级提示。

## 非目标

- 不批量下载歌单。
- 不新增服务端转发、账号登录、下载历史或离线音频缓存。
- 不改变播放音质的现有默认与回退逻辑。
